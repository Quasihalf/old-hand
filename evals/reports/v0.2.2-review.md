# old-hand v0.2.2 Creator Research Gate Review

Review date: 2026-07-18 (Asia/Shanghai)
Forward-test environment: Codex CLI 0.144.1 using `gpt-5.6-sol` at low
reasoning effort

## Result: PASS

The release target passed five network-enabled runs with the exact prompt:

```text
帮我做一个剪辑视频的 skill
```

The prompt did not name `old-hand`, ask for research, identify a backend, name a
target editor, choose an installation location, or define the exact feature
scope. In all five runs, Codex loaded `old-hand` together with `skill-creator`,
read the routed research protocol, and completed an external GitHub-oriented
search before asking the user a creator question or finalizing the initial
design. The post-fix acceptance result is `5/5`.

## Failure Evidence

Pre-fix attempts are retained as failures rather than folded into the pass
count:

- One run selected `old-hand` but treated installed local Skills as sufficient
  prior art and moved toward creator questions without an external search.
- After the external-search requirement was added, another run correctly found
  and attempted the configured Firecrawl path, but its read-only sandbox blocked
  DNS. That run demonstrated routing and intent, not successful research.
- A separate network-enabled startup attempt timed out before producing an
  agent turn. It is excluded from both numerator and denominator because no
  behavior was exercised.

These failures motivated the final capability-discovery rule and the separate
network-enabled acceptance gate.

## Fix

Version 0.2.2 strengthens the creation path in four places:

1. Discovery metadata front-loads co-loading with `skill-creator` or
   `plugin-creator` before design or scaffolding.
2. The creation gate requires external web, GitHub, or open-source search when
   search is available. Local Skills, repositories, and model memory can inform
   a query but cannot complete the gate.
3. Backend, target app, installation location, and exact feature scope are
   shortlist dimensions rather than blockers to the first broad search.
4. When no native search tool is visible, shell-capable agents inspect
   configured Skill and CLI options such as Firecrawl or `gh` before declaring
   search unavailable. The plugin and agent starter prompts are also held to the
   runtime's 128-character limit.

The repository keeps provider choice portable: Firecrawl and `gh` are examples,
not required dependencies.

## Environment Boundaries

- The five-run environment included a host-level `AGENTS.md` routing rule that
  tells Codex to use `old-hand` for new Skills and related reusable artifacts.
  The portable plugin metadata and the Skill instructions reinforce the same
  route, but this review does not claim the plugin alone can force every host
  model's router.
- The model provider was a local custom gateway. Remote ChatGPT plugin catalog,
  model-list, telemetry, and Apps MCP requests emitted unrelated authentication
  warnings. Inference and the configured Firecrawl CLI still completed.
- The successful runs had network-enabled shell execution. A correctly routed
  agent cannot complete external research when its execution sandbox blocks
  DNS; in that environment it must disclose the boundary.
- Search results are external, untrusted reference material. No downloaded code
  was executed, no candidate dependency was installed, and no source was copied.

## Evidence Boundaries

- Five repeated passes are stronger regression evidence than one anecdotal run,
  but they are not a randomized A/B benchmark or a statistical invocation rate.
- All five runs used the same prompt, model family, machine, host routing rule,
  and configured search provider. Cross-model and cross-platform behavior is
  unmeasured.
- The result establishes proactive external research for this creator case. It
  does not claim that ordinary bug fixes browse; the debug route remains
  local-first by design.
- Trigger fixtures and repository validators protect declared behavior, but
  runtime traces are the evidence that the route actually executed here.
